# QLanalyser 测试与验收详细设计

Date: 2026-06-22
Scope: 项目工作台前端、后端接口、EDF 上传链路、数据准备、项目管理、个人中心收口、端到端分析验收、评审系统多角度复核。
Boundary: 本文档定义测试与验收方法，不替代功能设计与模块算法设计。

## 1. 设计目标

测试与验收的目标不是“证明页面存在”，而是证明真实用户能够完成关键任务，并且每一步都有可追踪证据。

本方案要验证的核心事实是：

1. 用户能进入系统。
2. 用户能选择或创建项目。
3. 用户能上传或接入真实 EDF 文件。
4. 用户能在项目内看到文件、状态和准备信息。
5. 用户能完成数据预览、坏道/坏段处理、备注和准备确认。
6. 用户能进入后续分析并得到结果或明确失败原因。
7. 用户能下载结果或报告。
8. 评审系统能够从多个角度重复验证这些结论，避免假通过。

## 2. 验收原则

### 2.1 真实路径优先

所有验收都优先走真实 UI 点击路径，其次才使用 API 或脚本做状态核验。

### 2.2 结果优先

验收时不能只看按钮是否存在，必须看：

- 是否发生真实状态变化
- 是否生成真实产物
- 是否写入真实记录
- 是否能回读证据

### 2.3 失败必须可解释

失败时必须给出可执行的失败原因，例如：

- 按钮无响应
- 前端未传参
- 后端拒绝
- 资源不存在
- 状态冲突
- 乱码
- 入口层级错位

### 2.4 不接受假通过

以下情况都不能算通过：

- 页面打开了但功能没有真正执行
- 只有 API 成功，UI 没反馈
- 只写了说明，没生成证据
- 只验证了单个按钮，没验证整条路径
- 只看默认状态，没看错误状态

## 3. 验收分层

### 3.1 单元验收

目标：验证最小逻辑单元正确。

覆盖对象：

- 状态映射函数
- 文案清洗函数
- 项目/数据状态计算
- 准备方案 revision 冲突检测
- 文件备注显示规则
- 个人中心信息映射

### 3.2 接口验收

目标：验证前后端契约一致。

覆盖对象：

- 项目 CRUD
- 数据文件 CRUD
- EDF 上传
- 元数据读取
- 准备方案保存与读取
- 预览片段保存与读取
- 任务创建与状态查询
- 报告包下载

### 3.3 UI 验收

目标：验证用户通过点击能完成任务。

覆盖对象：

- 项目选择
- 项目进入
- 数据上传
- 数据展开
- 数据预览
- 准备确认
- 分析提交
- 结果查看
- 报告下载

### 3.4 端到端验收

目标：验证 EDF 从上传到结果的闭环。

覆盖对象：

- 上传真实或合成 EDF
- 选择项目
- 预览数据
- 标记坏道/坏段
- 保存准备方案
- 提交分析任务
- 查看结果
- 下载报告或结果包

### 3.5 评审系统验收

目标：验证同一条链路在多角色、多轮、不同观察角度下都成立。

覆盖对象：

- 多角色点击复核
- 结果截图/视频/日志
- 失败路径
- 口径一致性

## 4. 测试对象与顺序

测试顺序必须遵循：

1. 单元测试
2. 接口测试
3. UI 点击测试
4. 端到端验证
5. 评审系统多轮复核

顺序不能倒置。
如果前一层失败，下一层不得跳过，只能在修复后重测。

## 5. 前端验收设计

### 5.1 项目管理页

必须验证：

- 未选项目时不展开项目外数据
- 项目卡片能被正确选中
- 项目状态显示有业务意义
- 项目管理动作可理解
- 不相关入口不抢首屏

### 5.2 数据管理页

必须验证：

- 进入项目后才显示当前项目数据
- 数据状态显示可理解
- 重命名、更新、删除动作有明确反馈
- “替换文件”这类不自然表达不直接暴露给用户
- 没有上下文时不出现无效操作

### 5.3 数据准备层

必须验证：

- 数据预览真实可见
- 坏道、坏段、注释处理可操作
- 保存准备方案后状态变化正确
- 版本冲突会被提示
- 预览只影响当前准备上下文，不改写原始文件

### 5.4 个人中心

必须验证：

- 余额、充值、发票、账单、通知、安全设置可被访问
- 这些信息不污染主工作区
- 主流程和账户信息分层清晰

## 6. 后端验收设计

### 6.1 项目与数据

必须验证：

- 项目可创建、编辑、归档、删除
- 文件能归属到项目
- 项目状态可计算并返回业务标签
- 文件状态可计算并返回业务标签

### 6.2 准备方案

必须验证：

- `DataPreparationPlan` 能保存
- `revision` 会递增
- base revision 过期时能报冲突
- 公共准备信息能被后续分析引用

### 6.3 预览片段

必须验证：

- 预览片段能保存
- 预览片段能列出
- 预览片段能单独回读
- 预览片段能删除
- 预览片段包含可复核的窗口证据

### 6.4 错误与恢复

必须验证：

- 不支持格式有明确错误码
- 空输入有明确错误码
- 越界窗口有明确错误码
- 冲突有明确错误码
- 恢复建议可读

## 7. EDF 端到端验收设计

### 7.1 标准路径

1. 登录进入系统
2. 选择项目
3. 上传 EDF
4. 确认文件已进入当前项目
5. 打开预览
6. 检查数据与元数据
7. 标记坏道/坏段/注释处理
8. 保存准备方案
9. 提交分析任务
10. 查看结果
11. 下载报告

### 7.2 必测异常

必须同时验证至少以下异常之一：

- 空文件
- 不支持格式
- 无事件但走 ERP
- 未准备就提交分析
- 版本冲突
- 文件选择后不上传

### 7.3 验收输出

每轮端到端测试必须写出：

- 执行人
- 时间
- 环境
- 输入文件
- 测试路径
- 成功或失败原因
- 截图或日志
- 最终结论

## 8. 评审系统多角度复核设计

### 8.1 多角色

建议至少使用以下角色复核：

- 产品 owner
- 前端 reviewer
- 后端 reviewer
- 测试 reviewer
- 科学/方法 reviewer
- 发布/验收 reviewer

### 8.2 多轮

建议至少 5 轮：

1. 结构审查
2. 交互审查
3. 数据与状态审查
4. 端到端路径审查
5. 失败与边界审查

### 8.3 多视角

每一轮需分别看：

- 页面是否克制
- 信息是否有意义
- 功能是否真实
- 失败是否可解释
- 证据是否完整

## 9. 通过判定

### 9.1 必须满足

- 单元测试通过
- 接口测试通过
- UI 点击通过
- 端到端路径通过
- 评审系统多轮复核通过
- 关键证据可回读

### 9.2 条件通过

如果存在非主路径问题，但主路径可完整完成，可给条件通过，前提是：

- 问题已记录
- 修复计划明确
- 风险不影响主交付

### 9.3 失败

以下情况直接失败：

- 主路径断裂
- 结果不可回读
- 乱码或错误文案影响理解
- 功能假通
- 关键按钮无效
- 证据不完整

## 10. 验收产物

每次验收必须产出以下证据之一或多个：

- 评审报告
- 截图
- 浏览器录屏
- JSON 结果
- 日志
- 报告包
- 失败复现记录

## 12. 文档与代码交付要求

测试验收方案落地时，必须同时补齐：

- 前端验收检查点
- 后端验收检查点
- 端到端测试脚本
- 评审系统输出格式
- 失败回退路径

## 13. 最终验收口径

最终验收不是“功能大致齐了”，而是：

**真实用户能够用真实 EDF 数据，通过真实点击和真实后端链路，完成项目管理、数据接入、数据准备、任务提交、结果查看与报告下载，并且评审系统能多轮复核这些结论。**
