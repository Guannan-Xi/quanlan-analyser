# QLanalyser 产品级 UX 修复规格：把功能堆叠改成可操作的数据工作台

更新时间：2026-06-25
适用范围：QLanalyser Online 前端、Module Lab、数据管理、数据准备、分析任务、报告交付、管理员运营页。
触发来源：用户对数据管理、数据准备、波形预览、坏道/片段/事件编辑截图的集中反馈。
本文件不是提示词。它是后续修复代码、设计评审、测试验收必须读取的产品规格。

---

## 1. 一句话目标

把 QLanalyser 从“按钮很多、说明很多、用户要猜下一步”的页面，改成“点数据就看数据，点波形就修数据，所有修改能撤销，系统记录自动生成”的科研数据工作台。

用户不应该先理解内部流程名，也不应该为了看一个波形去找按钮。用户上传或选择 EEG 文件后，页面应立刻进入预览、质控、修正和确认的连续操作。

---

## 2. 本次截图暴露的四类问题

### 2.1 信息重复，占据空间，但不帮助用户完成任务

截图位置：图 1 数据管理页右侧红圈。

表现：

- 顶部已经展示“当前项目、数据文件、当前数据、下一步”。右侧又出现“我的分析项目、数据文件、下一步”。
- 右侧区域占了很大面积，但没有新的操作价值。
- “请从左侧选择一份数据文件”这类文字没有必要做成大卡片。
- 页面视觉中心被无效说明占走，真正的数据列表和下一步操作反而弱。

判断：这是“信息卡片冒充工作区”。

修复原则：

- 一个页面只保留一处状态摘要。
- 只要用户已经能从列表看到数据数量、当前选择和状态，右侧就不要重复。
- 空状态可以短，但不能压过主要任务。
- 数据管理页的主要动作是：上传、选择、预览、进入准备。

建议改法：

- 删除右侧重复卡片。
- 数据行本身承担选择入口。
- 当前数据状态显示在数据行或顶部轻量状态条。
- 右侧空间如果保留，应作为“选中数据详情 + 预览入口”，而不是项目说明。

验收条件：

- 首屏能看到数据列表和主要操作。
- 页面上不出现两处含义相同的“当前项目 / 数据文件 / 下一步”。
- 未选择数据时，只显示一条简短空状态：`选择一份 EEG 数据，右侧会显示预览和质量提示。`

---

### 2.2 多余按钮让用户重复确认同一件事

截图位置：图 2 右上“运行质控预览”。

表现：

- 用户已经在左侧选择了数据。
- 中间波形区域仍然是空白，并提示用户再点“运行质控预览”。
- 这一步对用户来说是重复劳动。
- 如果质控预览是必经步骤，就不应该要求用户再按一次。

判断：这是“系统必做动作被设计成用户按钮”。

修复原则：

- 单击数据 = 选中数据 + 加载波形 + 启动基础质控。
- 按钮只保留给用户真正可选择的动作。
- 如果某个动作每次都必须做，就自动做。
- 如果动作耗时，用进度状态告诉用户，不要让用户猜是否要点。

建议改法：

- 移除主按钮“运行质控预览”。
- 数据被选中后自动调用预览和基础质控接口。
- 波形区域显示以下状态：
  - 未选择：请选择一份 EEG 数据。
  - 加载中：正在读取波形和基础质量信息。
  - 成功：显示波形、通道、采样率、时长、事件数、疑似坏道。
  - 失败：显示失败原因和“重新加载预览”。
- 只有失败或用户主动刷新时，才显示“重新生成预览”。

验收条件：

- 点击任意数据卡片后，不需要再点“运行质控预览”即可看到预览或加载状态。
- 页面没有把必经动作包装成主按钮。
- 失败时有明确恢复入口：`重新加载预览`。

---

### 2.3 波形和修正工具分离，用户必须滚动找工具

截图位置：图 3 数据段与标签编辑。

表现：

- 删除片段、恢复片段、添加标签、编辑标签出现在波形下方较远位置。
- 用户必须先看波形，再滚动找操作，再回忆要处理的时间段。
- 这些操作本质上依赖波形上下文，不应该离开波形。

判断：这是“上下文相关工具脱离对象”。

修复原则：

- 波形是数据准备页的主工作对象。
- 凡是依赖波形位置的动作，都应贴近波形。
- 片段、坏道、事件标签不应是孤立表单，而应在波形上可见、可点、可撤销。

建议改法：

- 数据准备页中间为波形工作区。
- 波形上方放紧凑工具条：
  - 缩放、平移、复位。
  - 时间窗、增益、通道数。
  - 标记坏道。
  - 标记剔除片段。
  - 添加事件。
  - 撤销。
- 波形右侧放“当前修改”面板：
  - 已标记坏道。
  - 已剔除片段。
  - 已编辑事件。
  - 可恢复项。
- 用户拖选时间段后，工具条出现上下文动作：`剔除这段`、`添加事件`、`取消选择`。

验收条件：

- 用户不滚动也能看到波形和主要修正工具。
- 用户拖选波形区间后能立即剔除或加标签。
- 已剔除区间在波形上可见。
- 已剔除区间能单独恢复。

### 2.4 内部流程按钮暴露给用户，标题和动作看不懂

截图位置：图 4 “预处理质控就绪检查”“事件分段设置”。

表现：

- “预处理质控就绪检查”像内部验收标题，不像用户任务。
- “保存坏道审计”“下载处理计划”更像系统记录，不是用户当前要做的事。
- “丢弃坏道修改”含义不清：是撤销本次？恢复默认？删除所有记录？
- “保存事件分段”与“下载数据准备记录”“下载处理计划”混在一起，主次不分。

判断：这是“内部工作流泄漏到用户界面”。

修复原则：

- 用户界面说人话，内部记录自动做。
- 按钮只用于用户能理解、能决定、能撤销的动作。
- 系统生成的审计、处理计划、记录文件，不做成首屏大按钮。
- 下载类动作放到“处理记录”或“更多”。

建议改名：

| 当前文字 | 建议文字 | 备注 |
|---|---|---|
| 预处理质控就绪检查 | 数据质量检查 | 面向用户 |
| 运行质控预览 | 自动预览中 / 重新加载预览 | 不作为主按钮 |
| 检查数据基础信息 | 查看数据概况 | 如已自动显示，可删除按钮 |
| 确认数据准备 | 确认并进入分析 | 明确下一步 |
| 保存坏道审计 | 自动保存修改记录 | 不做主按钮 |
| 丢弃坏道修改 | 撤销本次坏道修改 | 必须说明范围 |
| 事件分段设置 | 事件与片段 | 更接近任务 |
| 保存事件分段 | 保存事件和片段修改 | 若自动保存，可降级 |
| 下载数据准备记录 | 下载处理记录 | 放更多菜单 |
| 下载处理计划 | 下载处理计划 | 放更多菜单 |

验收条件：

- 首屏主按钮不超过 3 个。
- 下载、审计、处理计划默认不出现在主流程按钮区。
- 所有撤销类按钮说明影响范围。
- 页面标题使用用户任务语言，而不是内部验收语言。

## 3. 全产品同类问题定义

后续修复不只处理这 4 张截图。所有页面都要扫描以下问题。

### A 类：重复信息区

典型表现：

- 同一页面多处显示同样的项目名、文件数、下一步。
- 卡片只说状态，不给操作。
- 空状态比主任务还显眼。

处理方法：

- 合并状态源。
- 删除重复卡片。
- 把状态压缩到顶部条、列表项或右侧详情。

### B 类：必经动作按钮化

典型表现：

- 用户选择对象后，还要再点“预览 / 检查 / 生成”。
- 系统明明可以自动执行，却把责任交给用户。
- 空白区域等待用户点按钮。

处理方法：

- 选择对象后自动加载。
- 失败或刷新才显示手动按钮。
- 耗时过程显示进度、取消、重试。

### C 类：工具离对象太远

典型表现：

- 看波形在上面，编辑片段在下面。
- 看报告在左边，解释按钮在另一个区。
- 操作前后用户要滚动或切页。

处理方法：

- 工具贴近对象。
- 对象被选中后出现上下文操作。
- 修改结果直接回显在对象上。

### D 类：内部流程暴露

典型表现：

- 页面标题像工程验收项。
- 按钮名像日志字段或后台任务。
- 用户不知道点击后会发生什么。

处理方法：

- 改成用户任务语言。
- 系统记录自动保存。
- 审计、下载、日志进入次级区域。

### E 类：不可恢复的破坏性操作

典型表现：

- 删除片段后看不到删除范围。
- 标坏道后不能恢复。
- 编辑事件后没有原值。
- 撤销按钮不清楚撤销什么。

处理方法：

- 坏道、剔除、事件修改都要有 before/after。
- 每项修改可单独恢复。
- 提交前有修改摘要。
- 保存后有处理记录。

### F 类：按钮过多但主任务不清楚

典型表现：

- 一排 5 个以上大按钮。
- 主按钮、次按钮、下载按钮、记录按钮混在一起。
- 禁用按钮没有说明。

处理方法：

- 主按钮最多 1 个，辅按钮最多 2 个。
- 记录/下载放更多菜单。
- 禁用按钮必须说明原因。

## 4. 产品级页面扫描清单

### 4.1 数据管理页

目标：让用户上传、选择、进入预览。

必须改：

- 删除重复项目摘要卡片。
- 数据行可点击，整行即选择。
- 已选择数据要有明显状态。
- 数据行显示：文件名、格式、通道数、采样率、时长、状态。
- “预览并准备”可以保留为行内次操作，但点击行本身也应进入预览。

不该出现：

- 大面积“下一步”说明卡。
- 重复的“我的分析项目”。
- 只有说明没有操作的右侧卡片。

### 4.2 数据准备页

目标：点选数据后立即看波形，完成质量检查、坏道、片段、事件修改。

必须改：

- 数据选择后自动加载预览。
- 删除主按钮“运行质控预览”。
- 波形、坏道、片段、事件在同一屏。
- 坏道可标记、可恢复。
- 片段可拖选剔除、可恢复。
- 事件标签可新增、编辑、恢复。
- 右侧显示修改记录。
- 确认按钮改为“确认并进入分析”。

建议布局：

```text
顶部：步骤条 + 当前数据概况
左侧：数据队列
中间：波形工作区
右侧：质量提示 + 当前修改
底部或右上：确认并进入分析
```

### 4.3 分析方法选择页

目标：让用户选择方法，而不是理解内部模块。

扫描点：

- 是否有过多方法卡片但没有推荐。
- 是否把内部 module_name 暴露给用户。
- 是否把“运行任务”作为唯一入口，而没有解释输入、输出和适用条件。

修复方向：

- 方法按任务组织：频谱、事件相关、时频、连接、参考/CSD。
- 每个方法卡片显示：适合什么数据、会产出什么、不适合什么。
- 高级参数折叠，默认给安全参数。
- 运行前检查数据是否已准备。

### 4.4 Module Lab

目标：作为方法开发测试台，而不是最终客户工作台。

扫描点：

- 是否把开发/验收语言露给普通用户。
- 是否主屏卡片过多。
- 是否结果、参数、下载链接挤在一起。

修复方向：

- 明确标注“方法测试台”。
- 方法卡片保留专业参数，但要有简短解释。
- 参数回显默认可见，便于审查。
- 结果区按：任务状态、参数、图表、下载、日志分组。

### 4.5 结果审查页

目标：让用户知道结果能不能用、为什么。

扫描点：

- 是否只给下载链接，没有解释。
- 是否把所有图表堆在一起。
- 是否缺少数据来源、参数、质量边界。

修复方向：

- 每个结果有来源数据、方法、参数、质量提示。
- 图表旁边显示“如何读”和“不要怎么读”。
- 不做临床诊断暗示。
- 失败/低质量结果要给下一步建议。

### 4.6 报告交付页

目标：让用户确认报告内容、下载、转发或提交。

扫描点：

- 是否把内部包、附件、日志直接暴露。
- 是否报告状态不清楚。
- 是否下载按钮过多。

修复方向：

- 报告主入口：查看报告、下载报告、下载数据包。
- 处理记录、审计、原始日志放到“详情”。
- 明确报告边界：科研分析，不用于临床诊断。

### 4.7 管理员运营页

目标：运营能看客户、任务、消费、系统状态。

扫描点：

- 是否混入客户不需要的技术项。
- 是否订单、任务、系统状态没有层级。
- 是否把危险操作放成显眼按钮。

修复方向：

- 分为客户、任务、订单/开票、系统状态。
- 危险操作二次确认。
- 系统日志默认折叠。

## 5. 坏道、片段、事件的统一交互模型

### 5.1 坏道

用户动作：

- 点击通道名或通道行。
- 选择“标记为坏道”。
- 坏道在波形中变灰或带虚线。
- 右侧修改记录新增一条：`Fp1 已标记为坏道`。
- 点击该记录的“恢复”，通道恢复。

数据记录：

```json
{
  "type": "bad_channel",
  "channel": "Fp1",
  "action": "mark",
  "source": "user",
  "reason": "manual_review",
  "created_at": "ISO_TIME",
  "reversible": true
}
```

验收：

- 标坏道后，前端状态、处理计划、任务参数都能看到该通道。
- 恢复后，该通道不再出现在坏道列表。
- 保存记录保留 before/after。

### 5.2 片段剔除

用户动作：

- 在波形上拖选时间区间。
- 点击“剔除这段”。
- 该区间出现半透明覆盖。
- 右侧修改记录新增：`12.0-18.0 s 已剔除`。
- 点击“恢复”，覆盖消失。

数据记录：

```json
{
  "type": "bad_segment",
  "start_sec": 12.0,
  "end_sec": 18.0,
  "action": "exclude",
  "source": "user",
  "reason": "manual_review",
  "reversible": true
}
```

验收：

- 剔除区间在波形上可见。
- 恢复后区间消失。
- 后续分析任务收到剔除区间。
- 不能出现结束时间小于开始时间。

### 5.3 事件标签

用户动作：

- 点击事件标记。
- 修改标签名或事件类型。
- 新增事件时，先选择时间点，再输入标签。
- 修改记录显示原标签和新标签。

数据记录：

```json
{
  "type": "event_label",
  "time_sec": 10.2,
  "before": "stimulus/1",
  "after": "target",
  "action": "rename",
  "reversible": true
}
```

验收：

- 标签修改后，波形标记同步变化。
- 修改可恢复。
- 导出的处理记录包含 before/after。

## 6. 按钮治理规则

### 6.1 主按钮

每个工作区最多一个主按钮。

数据准备页主按钮：

- `确认并进入分析`

### 6.2 次按钮

只保留用户会主动选择的操作：

- 重新加载预览。
- 撤销。
- 更多。

### 6.3 不应做成主按钮的动作

这些动作默认自动保存，或放到“更多 / 处理记录”：

- 保存坏道审计。
- 下载数据准备记录。
- 下载处理计划。
- 检查基础信息。
- 生成预览。
- 保存事件分段。

### 6.4 禁用按钮

禁用按钮必须说明原因，例如：

- `请先选择 EEG 数据`。
- `预览加载完成后可确认`。
- `没有可撤销的修改`。

## 7. 页面文字规则

### 7.1 写用户看得懂的话

使用：

- 数据质量检查。
- 当前数据。
- 波形预览。
- 已标记坏道。
- 已剔除片段。
- 修改记录。
- 确认并进入分析。

避免：

- 预处理质控就绪检查。
- 保存坏道审计。
- 事件分段设置。
- 数据准备记录生成。
- 运行质控预览。

### 7.2 科学边界

必须保留：

- 科研分析。
- 数据质量提示。
- 方法参数。
- 不用于临床诊断。

不能写：

- 诊断。
- 治疗建议。
- 精准定位脑区。
- 证明因果。
- 自动判定疾病。

## 8. 实施顺序

### 第 1 阶段：数据准备页闭环

范围：数据管理页 + 数据准备页。

必须完成：

1. 删除重复右侧项目说明卡。
2. 点击数据自动进入预览。
3. 移除“运行质控预览”主按钮。
4. 波形和坏道/片段/事件工具放同一屏。
5. 坏道可标记、恢复。
6. 片段可剔除、恢复。
7. 修改记录可见。
8. 确认并进入分析。

验收：见测试计划文件。

### 第 2 阶段：所有客户工作流页面扫描

范围：分析方法、结果审查、报告交付。

必须完成：

1. 删除重复状态区。
2. 降级内部记录按钮。
3. 改写内部流程标题。
4. 检查主按钮数量。
5. 加入失败/空/加载状态。

### 第 3 阶段：Module Lab 与管理员页

范围：Module Lab、管理员运营页。

必须完成：

1. 保留专业能力，但明确用户角色。
2. 开发/测试/验收语言不得进入普通客户页。
3. 管理员危险操作收敛到二级入口。
4. 下载、审计、日志折叠。

## 9. 代码实现注意事项

### 9.1 不要只改文案

这类问题不是文案问题。必须改交互关系：

- 自动触发。
- 就地编辑。
- 可恢复。
- 修改记录。
- 主次按钮分层。

### 9.2 不要做假功能

如果坏道恢复还没有后端支持，前端不能假装已保存。可先做本地 UI 状态，但必须标注“未保存”，并阻止进入正式分析。

### 9.3 不要破坏现有后端合同

若已有接口支持：

- `bad_channels`
- `bad_segments`
- `event_mapping`
- `parameters_json`
- `processing_plan`

优先复用。不要为 UI 改造另造一套孤立数据结构。

### 9.4 所有修改都要能导出处理记录

处理记录至少包含：

- 原始文件。
- 采样率、通道数、时长。
- 坏道修改。
- 片段剔除/恢复。
- 事件标签修改。
- 操作时间。
- before/after。
- 是否进入分析。

## 10. 通过标准

产品级通过不是“页面能点”。必须满足：

- 用户选数据后自动看到预览。
- 用户不滚动也能做主要修正。
- 坏道和剔除能操作、能恢复。
- 内部记录不压过主任务。
- 页面文字像产品，不像工程验收单。
- 所有关键状态有测试证据。
- 截图检查没有横向溢出、遮挡、主按钮过多。
