# QLanalyser CRO Traceability Contract

Date: 2026-06-20
Scope: research EEG analysis delivery and paid-pilot operations.

This contract defines the minimum traceability layer needed for QLanalyser to
evolve toward CRO-grade research delivery. It is not a claim of GxP, medical, or regulatory certification.

## Product Boundary

QLanalyser is a research EEG analysis tool and CRO infrastructure. It is not
medical software and must not provide clinical diagnosis, treatment advice,
medical-device claims, or clinical decision support.

## Traceability Question

For every delivered result, QLanalyser must be able to answer:

```text
Who did what, when, with which source data, which parameters, which software
version, which module version, which artifacts, which report package, which
account/order, and which reviewer or admin action released it?
```

## Minimum V1 Trace Objects

V1 must preserve these trace objects:

- Account and session.
- Project.
- EEG file and metadata.
- Data-preparation or QC plan when used.
- Analysis task.
- Normalized parameters.
- Module contract/lifecycle state.
- Artifact manifest with hashes.
- Report package and ZIP.
- Billing order, wallet transaction, and usage record.
- Invoice request and issued invoice attachment.
- Inbox delivery record.
- Admin operation.
- Audit event.

## Audit Event Contract

Each audit event must be append-only from the product perspective and include:

```yaml
audit_trace_id:
organization_id:
project_id:
actor_user_id:
action:
object_type:
object_id:
status:
metadata_json:
created_at:
```

Minimum V1 actions:

- `account.registered`
- `eeg_file.uploaded`
- `data_preparation.plan_confirmed`
- `analysis_task.created`
- `analysis_task.completed`
- `artifact.created`
- `report.created`
- `billing.recharge_created`
- `billing.recharge_confirmed`
- `billing.wallet_debited`
- `invoice.requested`
- `invoice.issued`
- `inbox.attachment_downloaded`
- `admin.invoice_reviewed`

If an action is not yet implemented as a formal audit event, release evidence
must clearly state the gap and link to the current equivalent record.

## Data Integrity Principles

QLanalyser should design toward ALCOA+ style research data integrity:

- Attributable: actor and account are known.
- Legible: records and reports are readable.
- Contemporaneous: timestamps are captured when actions occur.
- Original: source EEG and derived outputs are distinguishable.
- Accurate: parameters, software versions, and hashes are recorded.
- Complete: failures and rejected tasks are retained.
- Consistent: schemas and timestamps are consistent.
- Enduring: report packages and manifests remain retrievable.
- Available: authorized users and reviewers can retrieve evidence.

This is a design principle, not a regulatory certification claim.

## Review and Approval Ladder

V1:

- Admin operations can issue invoices and inspect operations state.
- Reports are generated by task completion and package creation.
- Review/approval is represented by admin actions and evidence records.

V1.5:

- Add analyst review status for report packages.
- Add scientific reviewer status for stable-module outputs.
- Add immutable report version field.

V2:

- Link module lifecycle state and validation evidence to each report.
- Add reviewer checklist per module.

V3:

- Add controlled release workflow: analyst review -> scientific review ->
  admin/customer release.
- Add electronic signature placeholder architecture where legally appropriate.

## Report Package Traceability

Every report ZIP must include or link to:

- Source EEG file id and metadata summary.
- Task id and module id.
- Workflow id.
- Raw and normalized parameters.
- Software versions.
- Method description.
- Artifact manifest with hashes.
- Limitations and non-medical boundary.
- Billing/order reference when part of paid delivery.

## Role Boundary

Minimum roles:

- Customer: upload data, run allowed workflows, download own outputs, request
  invoices, read own inbox.
- Admin: inspect operational state, issue invoice attachments, review billing
  records, and support customers.
- Future analyst/reviewer: review analysis outputs and approve controlled
  report release.

Customers must not access other customers' wallets, invoices, inbox messages,
or report artifacts.

## Evidence Matrix

Traceability evidence must map claims to files, commands, or runtime outputs:

| Claim | Evidence |
| --- | --- |
| Audit events exist | `scripts/acceptance_audit_quota_contract.py` |
| Billing, invoice, inbox, RBAC flow exists | `scripts/acceptance_ops_billing_invoice.py` |
| Report ZIP has reproducibility files | `scripts/acceptance_report_zip_contract.py` |

## Non-Claims

Do not claim:

- GxP certification.
- Medical-device compliance.
- Clinical diagnosis validity.
- Electronic signature compliance.
- Formal CRO validation completed.

Safe claim:

```text
QLanalyser is designed toward CRO-grade research traceability and currently
has V1 paid-pilot evidence for account, upload, task, artifact, report, billing,
invoice, inbox, admin, and audit records. Formal CRO/GxP validation remains a future controlled-validation activity.
```
