# QLanalyser 项目工作台后端详细设计

Date: 2026-06-22
Scope: 项目管理、项目内数据管理、EDF 上传与文件状态、数据准备方案、预览片段、任务前置校验、账户与通知所需基础能力。
Boundary: 不包含方法本身的算法细节实现；算法逻辑由各分析模块详细设计文档单独描述。

## 1. 设计目标

后端要为项目工作台提供稳定的“真实数据 + 可追踪状态 + 可恢复操作”基础能力。
核心目标是：

1. 项目、数据、准备方案、预览片段都能持久化。
2. 前端能拿到业务意义明确的状态，而不是原始枚举直出。
3. EDF 文件上传后可进入后续预处理与分析链路。
4. 所有变更都有审计痕迹和版本约束。
5. 允许前端按上下文逐层展开，而不是一次性暴露全部能力。

## 2. 领域对象

### 2.1 Project

项目是最上层容器，承载：

- 项目名称
- 所属用户/组织
- 归档状态
- 当前项目摘要
- 数据文件列表
- 准备方案列表
- 任务入口引用

### 2.2 EegFile

数据文件对象承载：

- 原始文件路径
- 文件格式
- 通道与采样率摘要
- 文件状态
- 用户备注
- 内部审计字段
- 与项目的归属关系

### 2.3 DataPreparationPlan

准备方案是项目内“共同可复用”的公共上下文。
它记录：

- 当前文件可见的准备状态
- 坏道
- 坏段
- 注释处理
- 预览偏好
- 滤波预览配置
- 版本号 revision
- 冲突校验信息

### 2.4 PreviewSegment

预览片段是当前准备状态下的证据对象，承载：

- 原始窗口
- 展示窗口
- 过滤预览窗口
- 图形输出
- 预览时的选择理由
- 与 plan revision 的关联

## 3. 状态模型

### 3.1 项目状态

项目状态应从内部枚举映射为用户可理解的业务状态。

建议状态：

- `可进入`
- `需上传数据`
- `准备中`
- `已归档`
- `暂不可用`

状态计算依据建议来自：

- 是否存在至少一个数据文件
- 最近一次准备方案状态
- 是否有未完成的阻止项
- 是否已归档

### 3.2 数据状态

建议状态：

- `已上传`
- `已预览`
- `已准备`
- `需处理`
- `不可用`

状态计算依据建议来自：

- 文件元数据是否完整
- 是否已有准备方案
- 是否已有有效预览片段
- 是否存在明显异常或拒绝项

### 3.3 准备方案状态

建议状态：

- `草稿`
- `已保存`
- `有冲突`
- `已确认`
- `已归档`

## 4. API 设计原则

### 4.1 设计原则

- 项目 API 只负责项目级容器。
- 数据 API 只负责当前项目内的文件。
- 准备 API 只负责当前文件的公共预处理上下文。
- 预览片段 API 只负责窗口级证据与回放。
- 所有 API 返回都应包含业务状态，而不仅是存储状态。

### 4.2 建议接口族

#### 项目

- `POST /api/projects`
- `GET /api/projects`
- `GET /api/projects/{project_id}`
- `PATCH /api/projects/{project_id}`
- `POST /api/projects/{project_id}/archive`
- `DELETE /api/projects/{project_id}`

#### 数据文件

- `POST /api/eeg/upload`
- `GET /api/eeg/files/{file_id}`
- `GET /api/eeg/files/{file_id}/metadata`
- `PATCH /api/eeg/files/{file_id}`
- `DELETE /api/eeg/files/{file_id}`

#### 数据准备方案

- `POST /api/eeg/files/{file_id}/data-preparation-plan`
- `GET /api/eeg/files/{file_id}/data-preparation-plan`
- `GET /api/eeg/files/{file_id}/data-preparation-plans`

#### 预览片段

- `POST /api/eeg/files/{file_id}/preview-segments`
- `GET /api/eeg/files/{file_id}/preview-segments`
- `GET /api/eeg/files/{file_id}/preview-segments/{segment_id}`
- `DELETE /api/eeg/files/{file_id}/preview-segments/{segment_id}`

## 5. 数据上传与归属

### 5.1 上传原则

- EDF 上传后必须归属到当前项目。
- 上传成功后返回明确的 file id、project id、文件格式、采样率、通道数摘要。
- 上传不能只是文件落盘，必须同时产生数据库记录。

### 5.2 文件状态写入

上传完成时至少写入：

- 原始文件位置
- 格式
- 大小
- 采样率
- 通道信息摘要
- 归属项目
- 上传时间
- 上传者

### 5.3 备注字段

用户备注与内部审计字段必须分离：

- 用户备注：面向用户展示，简短自然
- 内部审计字段：仅用于审计与回放，不直接裸露给普通用户

## 6. 准备方案设计

### 6.1 方案职责

准备方案是后续分析共享的公共前置层，用于记录：

- 坏道
- 坏段
- 注释处理
- 通道类型确认
- 重命名映射
- 预览参数
- 滤波预览参数

### 6.2 版本控制

准备方案必须支持 `revision`，因为：

- 多人协作可能同时修改
- 预览片段保存可能改变公共前置状态
- 后续分析必须能指向明确版本

当写入时如果 base revision 过期，后端返回冲突信息，前端提示用户合并或另存。

### 6.3 公共与专属边界

准备方案只保存后续分析通用的公共内容，不保存方法专属参数：

- 公共内容：坏道、坏段、注释处理、通道重命名、通道类型确认
- 专属内容：PSD 参数、ERP epoch 参数、TFR 参数、PAC surrogate 参数等

## 7. 预览片段设计

### 7.1 作用

预览片段是用户“当前看到的波形窗口”的证据对象，用于：

- 审核
- 复核
- 教学
- 报告附录

### 7.2 产物

预览片段应返回：

- metadata json
- display data json
- raw window data
- filtered display json
- svg/png 图形

### 7.3 原始文件原则

- 原始 EEG 文件不应被改写。
- 过滤预览只作用于当前窗口。
- 保存片段不等于生成正式处理后 EEG 文件。

## 8. 状态与前端映射

后端应返回“用户能看懂”的状态字段，例如：

- `project_status_label`
- `data_status_label`
- `plan_status_label`
- `blocking_reasons`
- `next_action_hint`

前端不得把后端原始枚举直接照抄给用户。

## 9. 错误设计

### 9.1 错误原则

错误信息要满足：

- 可理解
- 可定位
- 可修复
- 可复现

### 9.2 建议错误类型

- 文件不存在
- 文件格式不支持
- 通道不存在
- 时间窗越界
- 预览窗口过长
- 通道数量超限
- 准备方案版本冲突
- 预览片段保存失败
- 权限不足

### 9.3 错误输出要求

错误响应建议返回：

- `code`
- `message`
- `detail`
- `recoverable`
- `suggested_next_step`

## 10. 账户与个人中心支持能力

后端需要支持个人中心的基础数据读取：

- 当前账号信息
- 余额
- 充值记录
- 发票申请状态
- 通知列表
- 安全状态

这些信息应与主任务流程解耦，避免污染项目页主链路。

## 11. 审计与可追踪性

所有关键写操作都要记录：

- actor
- project_id
- file_id
- before / after
- timestamp
- revision
- source action

至少覆盖：

- 上传
- 备注修改
- 重命名
- 归档
- 删除
- 准备方案保存
- 预览片段保存

## 12. 后端验收标准

- 上传 EDF 后能落到项目与文件记录。
- 项目列表能返回业务意义明确的状态。
- 数据列表只返回当前项目的数据。
- 准备方案支持 revision 与冲突检测。
- 预览片段可保存、可列出、可删除。
- 返回值不直接把内部字段当用户状态。
- 错误信息可读且可修复。
- 账户与个人中心信息可被单独读取。
