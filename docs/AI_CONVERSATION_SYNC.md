# QLanalyser Online 对话同步与开发依据规则

## 1. 结论

QLanalyser Online 的软件架构设计、模块详细设计、验收标准和跨对话开发依据，统一以本仓库 Markdown 文档为准。

- GitHub / 仓库文档：唯一开发依据。
- 飞书：评审、会议纪要、跨团队沟通窗口。
- 对话记录：用于提炼结论，不直接作为开发依据。

## 2. 信息流

```text
用户与 AI 对话
  -> 提炼：决定 / 架构 / 模块设计 / 验收 / 风险 / 待定问题
  -> 写入仓库文档
  -> 生成飞书可复制摘要
  -> 更新当前交接文档
  -> 必要时本地 commit，等待用户确认后再 push
```

## 3. 固定落点

| 信息类型 | 仓库落点 | 用途 |
| --- | --- | --- |
| 永久产品/架构决策 | `docs/DECISIONS.md` | 所有对话必须遵守的规则 |
| 当前项目状态 | `docs/PROJECT_STATUS.md` | 新对话恢复上下文 |
| 时间线记录 | `docs/TASK_LOG.md` | 追踪每次同步和实现 |
| 当前接力摘要 | `docs/AI_HANDOFF_CURRENT.md` | 复制给新对话的启动依据 |
| 架构设计 | `docs/architecture/*.md` | 软件架构、服务边界、数据流 |
| 模块详细设计 | `docs/modules/*.md` | QC/PSD/ERP/TFR/PAC/Connectivity 等模块输入输出、参数、验收 |
| 飞书同步摘要 | 由模板生成 | 供人工复制到飞书或后续自动同步 |

## 4. 同步原则

1. 不把整段聊天原样塞进文档，只沉淀已确认结论和可执行事项。
2. 每个设计结论必须写清：背景、决定、影响范围、验收标准、未决问题。
3. 多个 AI 对话同时开发时，先读 `docs/AI_HANDOFF_CURRENT.md`、`docs/PROJECT_STATUS.md`、`docs/DECISIONS.md`。
4. 飞书内容如果与仓库不一致，以仓库为准，并从仓库重新生成飞书摘要。
5. 不记录 API key、token、客户隐私、真实原始 EEG 数据或本地敏感日志。
6. 不自动 push；push 必须等用户明确确认。

## 5. 推荐使用口令

用户可以直接说：

- “同步本轮对话”
- “把这个结论固化到开发依据”
- “更新架构设计依据”
- “更新某某模块详细设计”
- “生成飞书同步摘要”

AI 应启用项目 skill：`qlanalyser-conversation-sync`。

## 6. 飞书同步口径

飞书建议只放短摘要：

- 本次确认了什么
- 影响哪些模块
- 开发需要按哪些文档执行
- 本轮不做什么
- 下一步谁来做什么

如果没有飞书 API 或工具，AI 只生成可复制文本，不声称已经写入飞书。
